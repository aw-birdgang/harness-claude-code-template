【문서 목적】
  본 문서는 **지금 제품 모노레포나 앱 코드를 이 저장소 안에 생성하는 방법**을
  안내하는 것이 **아니다**.

  대신, 사용자가 **추후 하네스 엔지니어링**으로 실제 프로젝트를 진행할 때
  참고할 **초안(blueprint)** 으로 쓰인다:

    · 어떤 **타깃 스택·아키텍처 가정**(pnpm 모노레포, Nest/Next/React admin/RN,
      feature-base + Clean Architecture, 스택별 관용 컨벤션)을 두었을 때
    · **이 하네스 템플릿**이 무엇을 이미 제공하는지,
    · 하네스 쪽에 **무엇을 더 설계·작성·검증해야** 에이전트/사람 모두가
      같은 방향으로 워크플로를 굴릴 수 있는지

  를 정리한다.

  즉 “레포 트리 만드는 레시피”가 아니라 **하네스 역량 백로그·갭 분석**이다.


────────────────────────────────────────────────────────────────────────────────
0. 용어 정리
────────────────────────────────────────────────────────────────────────────────

  하네스(harness)
      Claude Code 등에서 agents / skills / commands / rules / hooks /
      manifests / 검증 스크립트로 구성되는 **AI 보조 개발 골격**.

  타깃 제품 스택(가정)
      사용자가 나중에 진행할 프로젝트의 기술 선택지. 이 저장소 바깥 또는
      별도 레포에서 실현된다고 가정한다.

  하네스 엔지니어링
      위 타깃에 맞춰 rules·skills·agents·매니페스트·훅·CI 검증 등을
      **설계하고 채워 넣으며 진화**시키는 활동.


────────────────────────────────────────────────────────────────────────────────
1. 이 저장소(템플릿)가 하는 일 — 한 줄 요약
────────────────────────────────────────────────────────────────────────────────

  Claude Code용 **소형·Node 전용** 하네스 **골격(skeleton) 템플릿**이다.

  실제 애플리케이션 코드(API, 웹, 모바일)는 **범위에 포함하지 않는다**.
  에이전트·스킬·슬래시 커맨드·규칙·매니페스트·훅 설정·경량 검증이
  **AI 보조 개발 워크플로의 기반**을 이룬다.

2. 현재 코드베이스 맵 (하네스 구성 요소별)
────────────────────────────────────────────────────────────────────────────────

2.1 루트 메타데이터
  - package.json
      * 패키지명 harness-claude-code-template, private, Node >= 20.
      * scripts: `npm test` → tests/run-all.js, `npm run plan` → install-plan-stub.js,
        `npm run harness` → scripts/ecc.js (bin: harness).
      * **역할**: 이 패키지는 “하네스 패키지”이며, 타깃 모노레포의 패키지 매니저와
        **직접 연동되지 않는다**(추후 하네스만 pnpm으로 관리할지는 팀 선택).

  - CLAUDE.md / AGENTS.md — 워크스페이스에서 쓸 명령·경로 안내.
  - README.md — minimal/core 프로파일 요약.
  - HARNESS_NODE_CONDENSE_STRATEGY.txt — 동형 축소 전략·Phase 로드맵.
  - .gitignore — 일반적 제외 항목.
  - .claude-plugin/plugin.json — 플러그인 메타(skills/commands 경로 등).

      아직 없음 → **초안 백로그: 스택별 또는 “경계 검사” 에이전트 정의**.


2.3 skills/
  - skills/harness-workflow/SKILL.md — 하네스 확장 절차(캐논 경로, manifests, hooks).
  【하네스 갭】 feature 슬라이스, Clean Architecture 레이어, 패키지 경계를

2.6 hooks/

3. 개발자가 지금 이 레포에서 쓰는 명령 (하네스 자체 검증)
────────────────────────────────────────────────────────────────────────────────

────────────────────────────────────────────────────────────────────────────────
4. 타깃 제품 스택(가정) vs 현재 하네스 커버리지
────────────────────────────────────────────────────────────────────────────────

  아래 “타깃”은 **미래 프로젝트 쪽 가정**이며, 이 템플릿에 그 구현이
  들어있어야 한다는 뜻이 **아니다**.

  구분 항목                     | 타깃 제품 스택(가정)          | 현재 하네스 템플릿
  ------------------------------|-------------------------------|---------------------------
  패키지/레포 레이아웃           | pnpm 워크스페이스 모노레포     | 단일 하네스 패키지 구조만
  백엔드                         | NestJS                        | 규칙에 한 줄 언급 수준
  프론트                         | Next.js                       | 동일
  어드민                         | React                         | 동일
  모바일                         | React Native                  | 동일
  구조 원칙                      | feature-base + Clean Arch    | 스킬·규칙에 미반영
  컨벤션                         | 스택별 관용 패턴             | common + TS 플레이스홀더

  해석: 타깃을 잘 지원하려면 **제품 레포를 여기서 만드는 것**이 아니라,
        하네스에 **규칙·스킬·에이전트·매니페스트·(선택) 검증·훅**을 채워
        “그런 레포에서 일할 때의 단일 진실”에 가깝게 만드는 **하네스 엔지니어링**
        가 필요하다.


────────────────────────────────────────────────────────────────────────────────
5. 하네스 엔지니어링 초안 — 무엇을, 어떻게 준비할지 (산출물 단위)
────────────────────────────────────────────────────────────────────────────────

  각 항목은 **이 저장소 안에서 작성·유지할 하네스 산출물**에 초점을 둔다.
  타깃 레포에 `apps/` 를 만드는 구체 명령어 나열은 **의도적으로 배제**한다
  (그건 제품 스캐폴딩·아키텍처 결정 단계에서 다룸).


5.1 아키텍처·경계를 하네스에 인코딩하기 (문서화 원칙)
  【목표】 feature-base + Clean Architecture를 **에이전트가 매번 재발명하지 않게**
      고정된 참조로 둔다.

  【하네스 산출물 예시】
  - skills/monorepo-feature-slice/SKILL.md (가칭)
        · 새 기능 작업 시 domain → application → infrastructure → presentation
          순으로 무엇을 검토할지 체크리스트.
        · “한 번에 하나의 feature 디렉터리” 우선 같은 **변경 단위 규칙**.
  - rules/architecture/clean-architecture-boundaries.md (가칭)
        · 허용되는 의존 방향, 금지 import 패턴을 **프로덕트 중립적** 서술
          (실제 경로 접두어는 타깃 레포 도입 후 규칙에 채워 넣음).

  【비고】 디렉터리 이름(`features/`, `modules/` 등)은 타깃 레포 합의사항이다.
      하네스 문서에는 **플레이스홀더 + 예시**만 두고, 팀 표준 확정 후 규칙을 갱신.


5.2 스택별 관용 컨벤션 — rules/ 확장 (하네스의 핵심 책임)
  【목표】 Nest / Next / React(admin) / RN 각각에 대해 AI와 사람이 같은 기준을 본다.

  【하네스 산출물 예시】(파일명은 예시)
  - rules/typescript/nestjs.md — 모듈 책임, DI, DTO·검증, 컨트롤러 두께.
  - rules/typescript/nextjs.md — RSC/클라이언트 경계, 데이터 패칭, env 접두.
  - rules/typescript/react-admin.md — 폴더 관습(features 등), 상태·API 레이어.
  - rules/typescript/react-native.md — 플랫폼 분기, 리스트 성능, 번들/네이티브 이슈.

  【운영 팁】 실제 ESLint/tsconfig 파일은 **타깃 레포**에 둔다. 하네스 규칙에는
      “어떤 설정 파일을 진실로 삼는지” 한 줄 링크·경로 패턴을 적어 두면
      에이전트가 코드와 문서를 함께 맞추기 쉽다.


5.3 에이전트 역할 분해 (하네스 엔지니어링)
  【산출물】 agents/*.md
  - planner 외에 예: 아키텍처 적합성만 보는 에이전트, 특정 스택 리뷰어 등.
  【원칙】 한 에이전트에 모든 스택을 몰아넣기보다, **좁은 책임 + SKILL 링크**가
      동형 축소 전략(§2.3 참고)과도 잘 맞는다.


5.4 슬래시 커맨드 — 반복 워크플로 고정 (선택)
  【산출물】 commands/*.md
  - 예: “기능 슬라이스 시작”, “플랜 확정 후 구현 단계 진입” 등 **팀 합의 절차**를
      한 페이지로 고정.


5.5 manifests — “어떤 하네스 번들을 쓸지” 프로파일 확장
  【목표】 minimal/core 외에 `fullstack-dev` 같은 프로파일로
      rules·agents·skills 모듈 묶음을 **의도적으로 선택** 가능하게 한다.

  【주의】 paths는 **이 하네스 저장소 안의 실제 경로**만 넣는 것이 안전하다.
      타깃 레포 경로를 paths에 적으면 install-apply가 없는 한 **의미만 있고**
      검증기와 어긋날 수 있다. “제품 레포 그래프”는 문서나 SKILL로 다룬다.


5.6 schemas + validate-skeleton 고도화 (하네스 품질)
  【산출물】 schemas/*.schema.json, validate-skeleton.js 단계 추가.
  【효과】 매니페스트·플러그인 JSON 형식 드리프트 방지.


5.7 hooks (선택, 프로파일 core)
  【산출물】 scripts/hooks/*, hooks.json 매핑.
  【용도】 금지 경로, 커밋 메시지 형식, 세션 시작 시 체크리스트 출력 등.


5.8 MCP·외부 지식 연동 (선택)
  【산출물】 mcp-configs 예시, (하네스 README의) “타깃 레포에 복사하는 법”.


5.9 ecc CLI 확장 (장기, 전략 문서 §2.6·§4와 정렬)
  install / doctor 등 — **하네스 배포 방식이 커졌을 때** 검토.


────────────────────────────────────────────────────────────────────────────────
6. 하네스 엔지니어링 백로그 체크리스트 (초안용)
────────────────────────────────────────────────────────────────────────────────

  제품 레포 생성 항목은 제외하고, **하네스 측에서 채울 일**만 나열한다.

  [ ] 타깃 스택 가정을 문서 상단에 한 페이지로 고정(버전·관리 패키지 매니저까지)
  [ ] architecture + feature-slice 를 다루는 SKILL 1개 이상 초안
  [ ] Nest / Next / React-admin / RN 용 rules/*.md 초안 (팀 표준 확정 후 정밀화)
  [ ] 필요 시 스택·역할별 agent 정의 초안
  [ ] 반복 워크플로용 command 초안 (선택)
  [ ] manifests 프로파일(예: fullstack-dev) 및 modules 정의 초안
  [ ] JSON Schema로 manifests 검증 도입 검토
  [ ] hooks 필요 여부 결정 후 scripts/hooks 초안 또는 “의도적으로 비활성” 기록
  [ ] MCP 연동이 있으면 mcp-configs + 하네스 README 한 절
  [ ] HARNESS_NODE_CONDENSE_STRATEGY.txt Phase 항목과 본 초안 우선순위 정렬


────────────────────────────────────────────────────────────────────────────────
7. 이 문서를 쓰는 방법 (추천 워크플로)
────────────────────────────────────────────────────────────────────────────────

  1) 타깃 제품의 아키텍처·폴더 합의가 어느 정도 잡히면, §5의 rules·SKILL에
     **실제 접두 경로·금지 의존**을 구체 숫자 없이라도 반영한다.
  2) 하네스만 먼저 `npm test`로 골격을 지키고, 매니페스트·스키마를 늘리면서
     검증기를 함께 키운다.
  3) 실제 코딩 레포는 별도이므로, **하네스 저장소와 제품 저장소를 같은 mono에 둘지,
     서브모듈·복사·플러그인 패키지로 배포할지**는 조직 표준으로 결정한다
     (본 초안은 그 결정을 강제하지 않음).
